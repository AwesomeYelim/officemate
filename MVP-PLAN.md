# OfficeMate MVP 개발 계획 (STEP 2)

> 기간: 2026-07-14 ~ 08-21 (6주, 근무시간 10%) · 최종 발표: 09-01 지란지교패밀리 데이
> 목표 한 줄: **"오피스챗에 '내일 점심 매운 거 3명' 치면 실제로 모임이 만들어지고, 사내에서 N건이 실제로 열렸다"를 9/1에 말할 수 있는 상태.**

## 0. 원칙

1. **데모가 스펙이다** — 화면·문구·상태 전이는 라이브 데모(landing)가 정본. MVP에서 새로 디자인하지 않는다.
2. **관통 시나리오로 측정한다** — 진척은 컴포넌트 완성도가 아니라 "E2E 시나리오 몇 개가 실메신저에서 도는가"로 센다.
3. **계약 먼저** — FE↔BE 인터페이스(JSON 스키마)는 코드보다 먼저 리포에 커밋. 변경은 상대 승인 후.
4. **W6은 동결** — 마지막 주 기능 추가 금지. 파일럿 실사용 지표 수집만.

## 1. 분담 (FE 2 · BE 2)

| 역할 | 담당자 | 영역 |
|---|---|---|
| **FE-1** | (이름) | 사용자 웹앱 — 온보딩 · 탐색 피드 · 마이페이지 · 정산 화면 (오피스챗 웹뷰) |
| **FE-2** | (이름) | 봇 인터페이스(카드 · 버튼 콜백 · 선발신) + HR 콘솔 실물화 |
| **BE-1** | (이름) | 코어 — 방 상태머신(모집→확정/방폭→체크인) · 매칭 엔진(규칙) · 마감 스케줄러 · DB |
| **BE-2** | (이름) | 도메인 — LLM 자연어 파싱 · 포인트 원장(예치/몰수/환급) · 정산 API · SSO 연동 |

- 각자 영역 안에서는 수직 완결 (화면이면 그 화면이 쓰는 API 소비까지 책임).
- 인터페이스 경계: FE↔BE-1(방·참여), FE↔BE-2(원장·유저), BE-1↔BE-2(매칭↔파싱 결과).

## 2. 주차별 백로그 (위에서부터 자르면 안 되는 순서)

### W1 (7/14~) — 봇 관통 스파이크 + 계약 [전원]
- [ ] 오피스챗 봇: 메시지 수신(웹훅) → 버튼 카드 발송 → 클릭 콜백 수신 실증
- [ ] 오피스챗 웹뷰 열기 + SSO(유저 신원 전달) 실증
- [ ] 봇 선발신(유저가 말 걸기 전 발송) 가능 여부 확인
- [ ] 리소스 4종 JSON 스키마 커밋: 방(room) · 참여(participation) · 원장(ledger) · 유저(user)
- [ ] DB 스키마 v1 + 서버 뼈대 배포 (호스팅 위치 확정)
- **완료 정의**: 실제 오피스챗에서 봇 카드 버튼을 누르면 우리 서버 로그에 콜백이 찍힌다.

### W2 — 핵심 루프 전반: 개설 → 매칭 → 확정
- [ ] LLM 파싱: 자연어 한 줄 → {슬롯, 인원, 취향} (BE-2)
- [ ] 방 상태머신: 생성 → 모집 → 정원 충족 시 확정 (BE-1)
- [ ] 규칙 매칭 스코어 v1: 관심 태그 교집합 + 부서 다양성 (BE-1)
- [ ] 봇 카드: 개설 확인 · 모집 현황 · 확정 알림 (FE-2)
- [ ] 온보딩 최소판: 관심 태그 + 식사 취향 (FE-1)
- **완료 정의**: 팀원 4명이 실제로 점심 모임을 한 번 만들어 먹는다 (도그푸딩 시작).

### W3 — 루프 완결: 디파짓 → 체크인 → 환급
- [ ] 포인트 원장: 예치 / 몰수 / 환급 / 적립 (BE-2)
- [ ] 참여 시 디파짓 500P 예치, 2시간 전 취소 무페널티 (BE-1+BE-2)
- [ ] 방장 수동 체크인 (QR 없음) → 환급 +50P (FE-2, BE-1)
- [ ] n-빵 정산 v1: 총액 기록 → 참석자별 몫 표시 (가상 원장, 실송금 없음) (FE-1, BE-2)
- **완료 정의**: 모임 1건이 예치→체크인→환급까지 원장에 정확히 기록된다.

### W4 — 반복 사용 장치: 선발신 루프 + 마이페이지
- [ ] 하루 3번(아침/점심/저녁) 슬롯 푸시 발송 (FE-2, BE-1 스케줄러)
- [ ] 모임 1시간 전 리마인더 → 취소 시 대기자 자동 승급 (BE-1, FE-2)
- [ ] 마이페이지: 포인트 지갑 · 참여 이력 · 신뢰도(단순 가산) (FE-1)
- **완료 정의**: 아무도 봇을 먼저 부르지 않아도 하루 3번 모임 제안이 온다.

### W5 — 탐색 + 방폭·병합 + HR 최소
- [ ] 탐색 피드: 모집 중 방 목록 · 참여 신청 (FE-1)
- [ ] 마감 미달: 인접 슬롯 병합 제안 → 거절/무응답 시 방폭 + 전액 환급 (BE-1)
- [ ] HR 콘솔 **대시보드 1페이지만** 실데이터 연결 (참여 건수 · 참여율 · 노쇼율) (FE-2)
- **완료 정의**: 사내 파일럿에 내놓을 수 있는 상태 (스스로 신규 유저로 온보딩해보고 막히는 곳 없음).

### W6 (8/18~) — 동결 · 파일럿 · 발표
- [ ] 기능 추가 금지, 버그 수정만
- [ ] 계열사/팀 단위 파일럿 (목표: 실모임 10건+)
- [ ] 실사용 지표 수집 → 발표 슬라이드 (참여 건수, 노쇼율, 재참여율)
- [ ] 9/1 발표 시나리오: 라이브 시연(자연어 개설) + 파일럿 지표 + 데모/목업으로 로드맵 커버

## 3. 스코프 구분 — "UI 포함·시뮬레이션" vs "로드맵"

**원칙: 전 기능이 UI에는 존재한다 (full package UI).** 실동작 깊이만 P0가 앞선다.

### 3a. UI까지 만들고 백엔드는 시뮬레이션 (파일럿에서는 "베타" 라벨 표시)
| 기능 | UI 범위 | 시뮬레이션 경계 |
|---|---|---|
| QR 체크인 | QR 표시·스캔 화면 | 실제 토큰 발급·검증 없음 (수동 체크인이 정식 경로) |
| 스폰서 시스템 | 스폰서드 배지·쿠폰 적용·스폰서 등록 화면 | 실제 과금·정산 없음 |
| 결제·송금 | 결제 수단 선택·송금·영수증 화면 | 가상 원장까지 (실money 이동 없음) |
| HR 콘솔 | 10페이지 전체 UI | 실데이터 연결은 대시보드 1페이지만, 나머지는 예시 데이터 |
| 온·오프 자동판별 | 판별 결과 표시 UI | 슬롯 길이 규칙만 (근무형태 연동 없음) |

### 3b. 로드맵 (UI도 구현도 없음 — 발표 로드맵 슬라이드로)
신뢰도 모델 학습(단순 가산만 사용) / 임베딩 매칭(태그 규칙만) / 잔디·네이버웍스 확장

## 4. 밀렸을 때 자르는 순서 (뒤에서부터)

HR 콘솔(W5) → 방폭·병합(W5) → 탐색 피드(W5) → 마이페이지(W4)
**절대 지키는 것**: W2·W3 핵심 루프 + W4 선발신. (9/1 발표 성립의 최소 집합)

## 5. W1에 확보해야 하는 기술 미지수 (오피스챗 팀 협의 필요)

| # | 확인 사항 | 막히면 |
|---|---|---|
| 1 | 서드파티 봇 등록 · 웹훅 · 카드/버튼 UI · 클릭 콜백 | 제품 전제 붕괴 → 오피스챗 팀 협조로 뚫는 것이 STEP 2 과제 |
| 2 | 봇 선발신(먼저 말 걸기) + 단체방 자동 생성 | 리텐션 루프 재설계 (유저 발화 트리거로 전환) |
| 3 | 웹뷰 + SSO(신원 자동 전달) | 매직링크 등 대체 인증 검토 |
| 4 | 조직 디렉터리(부서·이름) API | 온보딩 자기신고로 대체 |
| 5 | LLM 호출 경로 (외부 API 허용 여부 · 사내 오피스에이전트 활용 가능성) | 사내 경로로 전환 |

→ 위 1~4는 전부 오피스챗 개발팀 협의 사안. **W1의 첫 액션은 코드가 아니라 오피스챗 팀 창구 확보.**

## 6. 운영 리듬

- **주 1회 데모 데이** (금요일 권장): 그 주의 "완료 정의"를 실메신저에서 시연. 안 되면 다음 주 백로그를 줄인다.
- **계약 변경 규칙**: JSON 스키마 수정은 PR로, 반대편 담당자 승인 필수.
- **배포**: main push → Vercel(웹앱 미러) + namo(정식) 자동. 서버는 W1에 정한 호스팅에 CI 연결.
