01 — 사업 개요

Executive Summary

OfficeMate는 Slack 전용 랜덤 매칭 도구(아기고래·Donut)가 구조적으로 진입할 수 없는 국내 오피스챗·잔디·네이버웍스 시장을 타깃으로, 오피스챗에 플러그인으로 무료 설치되는 모임 활성화 엔진 위에 포인트 판매 수수료(Model A) → 개인카드 연동 페이백(Model B) → 조직 인사이트 수익 구조를 쌓는다. 아래 표는 본문 각 절의 핵심 수치를 한 장으로 요약한 것이다 — 세부 근거는 부록B에 정리한다.

예산은 쓰이는데, 관계는 만들어지지 않는다

단절
0번

같은 층, 대화 없음

재택·유연근무 확산 이후 옆 부서 동료와 한 번도 밥을 먹지 않은 채 분기가 지나갑니다. 신입은 물어볼 사람을 찾지 못하고 조기 이탈합니다 — 1명 이탈 비용은 채용·교육 포함 수천만 원.

도구 공백
Slack 전용

기존 도구는 국내에 못 들어온다

아기고래·Donut 등 랜덤 매칭 도구는 전부 Slack 전용 — 오피스챗·잔디·네이버웍스를 쓰는 금융·공공·제조 조직에는 선택지가 없습니다. 게다가 랜덤 매칭은 노쇼에 무너집니다.

온라인 확정

온·오프 자동판별

30분 이하 슬롯은 무조건 온라인 — 이동시간이 대화시간을 넘는 비효율을 제거합니다. 혼합 근무팀은 근무형태 기반으로 자동 배정.

노쇼 방지

포인트 보류 + 방폭

예치금 없이, 참석이 확인된 모임에만 포인트를 지급합니다 — 노쇼하면 포인트를 받지 못해 자연스럽게 참석을 유도합니다. 정원 미달이면 인접 슬롯 병합 제안 후 자동 방폭 — 죽은 방이 쌓이지 않습니다.

이번 달 적립 320P
두 층이 서로를 키운다 — OfficeMate의 출발점은 점심 번개·영어 스터디·러닝크루·보드게임 같은 자발적 소모임입니다. 이 수평적 만남이 참여 습관과 포인트·모임 등급 데이터를 만드는 성장 엔진이고, 그 위에 회식·팀모임·본부 행사 같은 회사 공식 모임과 예산이 얹혀 수익을 만드는 수익 엔진이 됩니다. 하나의 매칭 엔진이 두 층을 모두 처리하며, 어느 쪽도 다른 쪽을 위한 들러리가 아닙니다.

AI가 일하는 세 지점

"AI 매칭"이라는 포장이 아니라, 층위마다 AI가 아니면 못 하는 일을 맡깁니다. 데이터가 없어도 되는 부분은 규칙으로 정직하게, 데이터가 쌓여야 하는 부분은 로드맵으로 나눕니다.

층위 1 · LLM · MVP

자연어 인터페이스

"목요일 점심에 매운 거 먹을 사람 3명 구해줘" 한 줄이면 LLM이 슬롯·인원·취향으로 파싱해 방을 만듭니다. 그룹 전체의 제약을 종합한 설명 가능한 장소 추천 사유와 공통 관심사 기반 아이스브레이킹 대화 스타터까지 — 봇의 UX 자체가 AI입니다.

층위 2 · 임베딩+규칙 · MVP→학습

하이브리드 매칭 엔진

MVP는 규칙 스코어(관심사 태그 교집합 + 부서 다양성 + 메이트 등급)가 뼈대. 여기에 프로필 자기소개 텍스트의 임베딩 유사도를 더해 태그가 안 겹쳐도 이어지는 콜드스타트를 해결합니다 — "주말마다 북한산 가요" ≈ "등산화 샀어요". 데이터가 쌓이면 재만남률·체크인율을 라벨로 매칭 모델을 학습합니다.

층위 3 · 데이터 인사이트 · Phase 2~4

조직 데이터 응용

매칭 엔진이 만드는 참여 데이터가 확장 기능을 전부 엽니다 — 참여 이력 클러스터링으로 동호회 자동 발굴, 프로필 임베딩+RAG로 사내 전문가 검색("이거 아는 사람?"), LLM 대화형 조직문화 서베이, 매칭 엔진 재사용으로 온보딩 버디·TF 자동 모집.

하나의 엔진, 세 겹의 가치 — 매칭 엔진(층위 1+2)을 MVP로 만들면, 거기서 나오는 참여 데이터가 층위 3의 응용을 전부 가능하게 합니다. 경쟁사가 기능을 베껴도 재만남률 라벨이 쌓인 매칭 모델은 데이터 없이는 복제할 수 없는 해자입니다.
04 — 리텐션 가설

3주 뒤에도 쓰는 이유 — 습관 루프를 설계하고, 숫자로 검증한다

이 카테고리의 진짜 사망 원인은 노쇼가 아니라 출시 3주 뒤의 침묵입니다. OfficeMate는 참여가 저절로 반복되도록 트리거 → 보상 → 자산의 습관 루프를 설계하고, 베타에서 아래 지표로 가설을 검증합니다 — 숫자가 안 나오면 루프를 고치는 것이 베타의 목적입니다.

트리거
하루 3번

봇이 먼저 말을 건다

아침·점심·저녁 슬롯이 매일 돌아오는 최고 빈도 트리거. 사용자가 기억해서 여는 앱이 아니라, 끼니마다 봇이 슬롯 카드를 보내는 푸시형 루프 — 방문 동기가 소진되지 않습니다.

같은 버튼을 눌러도 결과가 매번 다릅니다 — 오늘은 옆 부서의 등산 메이트, 내일은 파스타집 2만 원 지원 모임. 예측 불가능한 보상이 재참여를 만듭니다.

투자 자산
번개 → 제트

쌓일수록 올라가는 등급

모임을 만들거나 참여할 때마다 포인트가 쌓이고, 누적 횟수에 따라 메이트 등급이 오릅니다(번개→푸른번개→레드번개→자이언트 제트, 09절 로드맵). 등급이 오르면 보너스 포인트도 커져, 떠나면 잃는 자산이 생기는 순간 이탈에 비용이 붙습니다.

지급 포인트 크기의 균형을 알고 있다 — 참석해야만 포인트를 준다는 원칙만으로 노쇼는 억제되지만, 지급액이 너무 적으면 참여 유인 자체가 약해집니다. 그래서 베타에서 참석당 지급 포인트 액수(예: 100P / 300P / 500P)를 조직별로 달리 적용해 참여율–지급 규모 트레이드오프의 균형점을 데이터로 결정합니다.

06 — 비즈니스 모델

무료로 설치되고, 포인트가 오갈 때마다 수익이 쌓인다

가장 어려운 수익을 전제하지 않고, 가장 확실한 수익부터 순서대로 쌓습니다. 오피스챗에 플러그인으로 무료 설치돼 도입 장벽이 없고, HR이 모임 포인트를 구매·충전할 때 발생하는 판매 수수료(Model A)가 핵심 수익이며, 개인카드 연동 페이백(Model B)이 그 충전 전환을 끌어올리는 보조 장치가 됩니다. 참여 데이터에서 나오는 조직 인사이트가 그 위에 성장을 더합니다. 연도별 매출 규모는 07절 재무계획에서 별도로 다룹니다.

대상 고객 — 기업 HR 부서

임직원 간 모임과 교류를 활성화하고 싶지만 마땅한 수단이 없는 HR 담당자가 1차 고객입니다. OfficeMate는 조직문화 예산을 "실제 일어난 모임"에만 정확히 집행하게 합니다 — 02절의 "예산은 쓰이는데 관계는 만들어지지 않는다"는 문제에 대한 직접적인 답입니다.

돈의 사이클

HR이 충전한 포인트가 임직원의 모임 보상이 되고, 그 포인트가 다시 회식·모임 장소 결제로 흘러가는 하나의 순환입니다. 이 순환에서 포인트가 손을 바꾸는 지점(HR의 구매·충전)마다 판매 수수료가 곧 우리의 수익이 됩니다.

1
HR 부서포인트 구매·충전 — 이 시점의 판매 수수료가 Model A 매출
2
임직원모임 생성·참여 — 슬롯 예약부터 체크인까지 03절의 매칭 엔진을 그대로 사용
3
포인트 적립참석이 확인된 모임에만 지급, 모임 등급에 따라 보너스 — 노쇼하면 지급되지 않는다
4
장소에서 사용파트너 매장에서 포인트로 직접 결제하거나, 개인카드로 결제 후 영수증이 자동 연동돼 페이백을 받는다(Model B)
5
재충전모임 활성화 성과를 확인한 HR이 포인트를 추가 구매 — 다시 1로 순환
⚡ HR이 확보한 가용 포인트 안에서만 지급되므로, 예산이 통제된 상태로 모임 활성화가 이뤄집니다. 모임이 늘수록 포인트 소진이 늘고, 소진된 만큼 재충전이 일어나 Model A 매출이 사용량에 비례해 확장됩니다.

⚖️ 조직의 자금을 충전·예치받는 구조는 전자금융업·여신전문금융업 규제 검토가 선행 — 초기에는 PG·법인카드사 제휴 구조로 라이선스 부담 없이 시작하고, 충전 규모가 검증된 후 직접 등록을 판단한다(11절 리스크 참조).

② Model B

개인카드 연동 페이백 — 보조 장치

영수증 자동 연동

임직원이 개인 카드를 등록하고 모임 장소에서 결제하면 영수증이 자동 연동돼, HR이 충전한 포인트에서 일부가 차감되어 임직원에게 페이백됩니다. 임직원은 할인 혜택을, HR은 실사용 기반의 투명한 예산 집행을 얻습니다 — 별도 수수료 라인이 아니라 ①의 포인트 소진·재충전 속도를 끌어올리는 보조 장치입니다.

유사 카테고리 검증 — "참여하면 포인트가 쌓이고, 그 포인트가 실제 결제로 이어지는" 구조는 새로운 발명이 아니라 이미 검증된 카테고리의 국내·B2B 적용입니다. 해외 직접 comp로는 사내 참여·인정 보상 플랫폼인 Bonusly·Nectar·Kudos·Fringe가 있고(이들은 구독+포인트 모델이라 우리보다도 매출 바닥이 있는 형태), 포인트 발행 후 일부가 소진되지 않는 브레이키지·예치금 플로트 수익 구조는 스타벅스 리워드 같은 프리페이드 포인트 비즈니스에서 이미 상용화돼 있습니다. 국내에서는 당근마켓·토스처럼 낮은 수수료율 × 높은 거래 빈도로 이익을 내는 플랫폼 패턴이 이미 자본시장에서 검증됐습니다 — OfficeMate의 도전은 "새로운 모델을 증명하는 것"이 아니라 "검증된 패턴을 국내 오피스챗 채널에 얼마나 빠르게 이식하는가"에 가깝습니다.

10 — 미구현 항목 실현 계획

다섯 가지 중 넷은 데모에서 실동작까지 검증했고, 남은 인프라 연동만 순서대로 붙입니다

해커톤 데모(💬)는 핵심 매칭 루프에 더해 온·오프 자동판별·QR 체크인(①), 온보딩·메이트 등급/포인트·탐색 피드(②), 가상 포인트 원장(⑤)까지 실동작으로 구현했고, 랜딩(🤝)에서는 방폭 시나리오(④)를 검증했습니다. 다섯 항목 모두 UI·로직은 클라이언트 사이드로 완성됐고 서버·외부 인프라 연동만 남았습니다. 실제 메신저 연동(③)만 아직 API 실사 단계이며, 항목별로 무엇을 검증했는지와 실서비스 전환에 무엇이 필요한지를 명시합니다.

데모에서 실동작 검증 완료

① 온·오프라인 자동판별 · QR 체크인/출석 확인

현재 상태 — 온·오프 판별 규칙("30분 이하 슬롯은 온라인")은 본 계획서 03절에 정의돼 있고, 데모(💬)에서 체크인 카드가 이 규칙에 따라 온라인은 "입장 확인", 오프라인은 "QR 스캔 체크인" 버튼으로 자동 분기되는 것까지 클라이언트 사이드 로직으로 구현·검증했습니다. 다만 이는 시연용 상태 전이이며, 실제 QR 토큰 발급·화상회의 API 연동은 아직 없습니다.

  1. MVP수동 체크인 — 방장이 봇 카드의 "출석 확인" 버튼을 눌러 참가자를 체크한다. 위치 검증 없이 자기신고 기반으로 단순화해 QR 인프라 없이 우선 출시한다. 온·오프 자동분기 UI는 데모에서 검증 완료.
  2. 베타QR 체크인 — 방 확정 시 방별 서명된 단기 토큰으로 QR을 발급, 참가자가 인앱 카메라로 스캔하면 체크인 API가 호출된다. 오프라인 모임 전용.
  3. 정식출시화상 접속 확인 — 온라인 모임은 화상회의 참가자 목록 조회 API로 자동 확인(수동 확인 병행), 온·오프 판별은 근무형태 데이터가 있는 조직에 한해 자동화하고 없으면 슬롯 길이 규칙만 적용한다.

⚠ QR은 물리적 프린트·부착이 필요해 파일럿 초기엔 운영 부담이 크다. 자기신고 체크인으로 시작해 누적 참여 데이터가 쌓인 뒤 QR로 전환하는 순서를 권장한다.

데모에서 실동작 검증 완료

② 온보딩·프로필 등록 · 메이트 등급/포인트 · 탐색 피드

현재 상태 — 사용자 여정 목업(🧭)에 있던 온보딩·프로필·등급 링 화면을 데모(💬)에도 이식해, 최초 실행 시 온보딩 모달에서 관심사 태그·자기소개를 입력하면 등급 링이 채워지고 탐색 피드에서 누적 참여 기준 자동 승인까지 클라이언트 사이드 로직으로 재현했습니다. 메이트 등급(번개→제트) 산식과 탐색 피드의 서버 연동은 아직 없습니다.

  1. MVP최초 대화 온보딩 — 봇 최초 실행 시 관심사 태그 버튼 선택 + 자기소개 한 줄을 받아 프로필 DB에 저장하고, 자기소개 텍스트는 매칭엔진 층위 2 임베딩 파이프라인에 그대로 재사용한다. 데모에서 검증 완료.
  2. 베타메이트 등급 산식 — 누적 모임 생성·참여 횟수 기준으로 시작(번개 3회→푸른번개 5회→레드번개 10회→자이언트 제트 20회, 05·ROADMAP). 계산에 필요한 체크인 데이터가 베타 기간에야 쌓이므로 MVP 단계엔 배포할 수 없다. 데모의 등급 링은 온보딩·체크인 이벤트에 횟수를 더하는 단순 버전이다.
  3. 정식출시탐색 피드 — "이번 주 모집 중인 모임" 카드 리스트를 발송(부록에서 확인된 카드형 메시지 UI 활용), 메이트 등급을 정렬·필터 기준으로 노출한다. UI는 데모에서 검증 완료, 서버 연동은 별도.
계획서 조사만 존재

③ 실제 메신저(잔디·네이버웍스) 연동 · 예약/캘린더 실동작

현재 상태 — 부록A에서 3사 봇 API를 문서 기준으로 실사했지만, 실제 봇 앱 등록·Webhook 수신 서버·OAuth 인증 코드는 아직 없습니다. 데모·랜딩은 모두 브라우저 시뮬레이션입니다.

  1. MVP네이버웍스 봇 등록 — 개발자센터에서 봇 앱 생성 → Webhook URL 발급 → 콜백 수신 서버(자연어 파싱)와 OAuth 2.0 토큰 발급/갱신 로직을 구현한다. 실사 결과 자유 텍스트 콜백이 되는 유일한 채널이라 MVP는 네이버웍스 단일 타깃으로 좁힌다.
  2. 베타캘린더 자동 등록 — 확정된 모임을 참가자 캘린더에 등록(제목·시간·장소). 캘린더 API는 이번 실사 범위 밖이라 베타 착수 전 별도 실사가 필요하다.
  3. 충전검증네이버 예약 연동 — 파트너센터 심사가 전제라 MVP는 "예약 링크 제공"으로 축소하고, 실연동은 06절의 예약 제휴 커미션과 같은 시점에 착수한다. 카카오워크는 모달 폼 개설로 변형 구현, 잔디는 주최 측 API 협의가 전제된다.

⚠ 네이버웍스 봇은 방 생성 시점에만 멤버를 지정할 수 있어(부록A 실사 결과), 인접 슬롯 병합 시 방 재생성이 필요해 기존 설계보다 API 호출이 늘어난다.

데모에서 실동작 검증 완료

④ 방폭 (72시간 미달 자동 해산)

현재 상태 — 랜딩 데모(🤝)에서 타임어택 가속 버튼으로 실제 트리거해 "방폭 → 안내 메시지" 시퀀스가 상태머신으로 재현됨을 검증했습니다. 예치금이 없는 구조라 환급 절차는 필요 없고, 다만 이는 클라이언트 사이드 타이머로 만든 시뮬레이션입니다.

  1. MVP서버 사이드 타이머 — 방 생성 시 마감 시각(생성+72h)을 DB에 기록하고 배치 잡(cron 또는 지연 큐)이 주기적으로 미달 방을 스캔·해산한다. 데모의 상태 전이 로직을 그대로 서버로 이관한다.
  2. MVP인접 슬롯 병합 제안 — 방폭 확정 전 매칭엔진이 인접 슬롯 후보를 조회해 병합 제안 카드를 먼저 발송하고, 미응답 시에만 방폭을 확정한다.
  3. 정식출시해산 알림 · 로그 — 방폭 시 참가자 전원에게 해산 사유를 안내하고, 재발 방지를 위해 슬롯·시간대별 방폭률을 집계해 매칭 엔진 튜닝에 반영한다.
MVP(가상 포인트 원장) 실동작 검증 완료

⑤ 포인트 원장·개인카드 페이백(Model B)·스폰서십 · HR 대시보드 연동

현재 상태 — 스폰서 셀프서브 목업과 별도 HR 대시보드 목업(📊)은 여전히 실제 결제·전자결의·조직 데이터 파이프라인이 없습니다. 다만 Model A의 핵심인 가상 포인트 원장(HR 선결제 충전 → 모임 참석 확인 시 지급 → 장소 결제 시 차감)은 데모(💬)에서 포인트 잔액이 실제로 오가는 것까지 실동작으로 구현·검증했습니다. Model B(개인카드 연동 페이백)의 PG 결제 연동과 HR 대시보드 실연동은 아직 없고, "카드 자동 페이백" 문구는 여전히 목업 연출입니다.

  1. MVP가상 포인트 원장 — 개인카드 PG 실연동 대신, HR이 선결제로 포인트를 충전하고 모임 참석 확인 시 지급·장소 결제 시 차감되는 자체 원장으로 우회한다. 조직별 ERP·전자결의 시스템은 표준 API가 없어 커스텀 연동이 불가피하기 때문이다. 데모에서 검증 완료.
  2. 충전검증Model B 카드 연동 PG 연동 — PG사 결제 + 사업자 인증 API로 개인카드 등록·영수증 자동 인식을 구현하고, 결제 확인 시 포인트 원장에서 자동으로 페이백을 차감·지급하는 훅을 추가한다. 스폰서 셀프서브 온보딩도 같은 PG 연동 위에서 구현한다.
  3. 플랫폼확장 이후HR 대시보드 실연동 — 매칭엔진의 체크인·노쇼·부서조합 로그를 집계 배치로 대시보드 API에 공급한다. 조직 HRIS와의 SSO·구성원 매핑이 필요하나 부서 매핑은 조직도 API로 이미 확인됐다. 법인카드 직접 연동은 조직별 개별 계약 건으로 Enterprise 단계에서만 대응한다.
공통 패턴 — 다섯 항목 모두 UI·로직 설계는 끝났고 외부 인프라(서버 배치, 결제 PG, 캘린더/예약 API, 조직 HRIS)를 붙이는 일만 남았다는 공통점이 있습니다. ①②④⑤ 네 항목은 데모·랜딩에서 클라이언트 사이드 실동작 검증까지 마쳤고, 실메신저 연동(③)만 API 실사 단계로 남아 있습니다. MVP 범위(~2026.08)는 서버 사이드 방폭 배치, 체크인·온보딩·포인트 원장의 서버 API 이관, 네이버웍스 봇 등록까지이고, 나머지는 09절 실행 계획의 베타·정식출시·충전검증 단계에 맞춰 순서대로 붙입니다.
