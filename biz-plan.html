신뢰도 점수 92
두 층이 서로를 키운다 — OfficeMate의 출발점은 점심 번개·영어 스터디·러닝크루·보드게임 같은 자발적 소모임입니다. 이 수평적 만남이 참여 습관과 신뢰도 데이터를 만드는 성장 엔진이고, 그 위에 회식·팀모임·본부 행사 같은 회사 공식 모임과 예산이 얹혀 수익을 만드는 수익 엔진이 됩니다. 하나의 매칭 엔진이 두 층을 모두 처리하며, 어느 쪽도 다른 쪽을 위한 들러리가 아닙니다.

AI가 일하는 세 지점

"AI 매칭"이라는 포장이 아니라, 층위마다 AI가 아니면 못 하는 일을 맡깁니다. 데이터가 없어도 되는 부분은 규칙으로 정직하게, 데이터가 쌓여야 하는 부분은 로드맵으로 나눕니다.

층위 1 · LLM · MVP

자연어 인터페이스

"목요일 점심에 매운 거 먹을 사람 3명 구해줘" 한 줄이면 LLM이 슬롯·인원·취향으로 파싱해 방을 만듭니다. 그룹 전체의 제약을 종합한 설명 가능한 장소 추천 사유와 공통 관심사 기반 아이스브레이킹 대화 스타터까지 — 봇의 UX 자체가 AI입니다.

층위 2 · 임베딩+규칙 · MVP→학습

하이브리드 매칭 엔진

MVP는 규칙 스코어(관심사 태그 교집합 + 부서 다양성 + 신뢰도)가 뼈대. 여기에 프로필 자기소개 텍스트의 임베딩 유사도를 더해 태그가 안 겹쳐도 이어지는 콜드스타트를 해결합니다 — "주말마다 북한산 가요" ≈ "등산화 샀어요". 데이터가 쌓이면 재만남률·체크인율을 라벨로 매칭 모델을 학습합니다.

층위 3 · 데이터 인사이트 · Phase 2~4

조직 데이터 응용

매칭 엔진이 만드는 참여 데이터가 확장 기능을 전부 엽니다 — 참여 이력 클러스터링으로 동호회 자동 발굴, 프로필 임베딩+RAG로 사내 전문가 검색("이거 아는 사람?"), LLM 대화형 조직문화 서베이, 매칭 엔진 재사용으로 온보딩 버디·TF 자동 모집.

하나의 엔진, 세 겹의 가치 — 매칭 엔진(층위 1+2)을 MVP로 만들면, 거기서 나오는 참여 데이터가 층위 3의 응용을 전부 가능하게 합니다. 경쟁사가 기능을 베껴도 재만남률 라벨이 쌓인 매칭 모델은 데이터 없이는 복제할 수 없는 해자입니다.
02.5 — 리텐션 가설

3주 뒤에도 쓰는 이유 — 습관 루프를 설계하고, 숫자로 검증한다

이 카테고리의 진짜 사망 원인은 노쇼가 아니라 출시 3주 뒤의 침묵입니다. OfficeMate는 참여가 저절로 반복되도록 트리거 → 보상 → 자산의 습관 루프를 설계하고, 베타에서 아래 지표로 가설을 검증합니다 — 숫자가 안 나오면 루프를 고치는 것이 베타의 목적입니다.

쓸수록 좋아지는 계정

체크인마다 신뢰도 점수·포인트가 쌓이고 매칭 우선순위가 올라갑니다. 떠나면 잃는 자산이 생기는 순간, 이탈에 비용이 붙습니다.

04 — 비즈니스 모델

확실한 바닥, 열려 있는 천장 — 한 조직에서 세 층의 수익이 쌓인다

가장 어려운 수익을 전제하지 않고, 가장 확실한 수익부터 순서대로 쌓습니다. 검증된 좌석 구독(SaaS)이 예측 가능한 바닥을 깔고, 참여 데이터에서 나오는 조직 인사이트가 성장을 만들며, 모임 예산 거래는 거래액이 검증되는 만큼 열리는 업사이드입니다.

07 — 미구현 항목 실현 계획

다섯 가지 중 넷은 데모에서 실동작까지 검증했고, 남은 인프라 연동만 순서대로 붙입니다

해커톤 데모(💬)는 핵심 매칭 루프에 더해 온·오프 자동판별·QR 체크인(①), 온보딩·신뢰도 링·탐색 피드(②), 가상 크레딧 원장(⑤)까지 실동작으로 구현했고, 랜딩(🤝)에서는 방폭 시나리오(④)를 검증했습니다. 다섯 항목 모두 UI·로직은 클라이언트 사이드로 완성됐고 서버·외부 인프라 연동만 남았습니다. 실제 메신저 연동(③)만 아직 API 실사 단계이며, 항목별로 무엇을 검증했는지와 실서비스 전환에 무엇이 필요한지를 명시합니다.

데모에서 실동작 검증 완료

① 온·오프라인 자동판별 · QR 체크인/출석 확인

현재 상태 — 온·오프 판별 규칙("30분 이하 슬롯은 온라인")은 본 계획서 02절에 정의돼 있고, 데모(💬)에서 체크인 카드가 이 규칙에 따라 온라인은 "입장 확인", 오프라인은 "QR 스캔 체크인" 버튼으로 자동 분기되는 것까지 클라이언트 사이드 로직으로 구현·검증했습니다. 다만 이는 시연용 상태 전이이며, 실제 QR 토큰 발급·화상회의 API 연동은 아직 없습니다.

  1. MVP수동 체크인 — 방장이 봇 카드의 "출석 확인" 버튼을 눌러 참가자를 체크한다. 위치 검증 없이 자기신고 기반으로 단순화해 QR 인프라 없이 우선 출시한다. 온·오프 자동분기 UI는 데모에서 검증 완료.
  2. 베타QR 체크인 — 방 확정 시 방별 서명된 단기 토큰으로 QR을 발급, 참가자가 인앱 카메라로 스캔하면 체크인 API가 호출된다. 오프라인 모임 전용.
  3. 정식출시화상 접속 확인 — 온라인 모임은 화상회의 참가자 목록 조회 API로 자동 확인(수동 확인 병행), 온·오프 판별은 근무형태 데이터가 있는 조직에 한해 자동화하고 없으면 슬롯 길이 규칙만 적용한다.

⚠ QR은 물리적 프린트·부착이 필요해 파일럿 초기엔 운영 부담이 크다. 자기신고 체크인으로 시작해 신뢰도 점수가 쌓인 뒤 QR로 전환하는 순서를 권장한다.

데모에서 실동작 검증 완료

② 온보딩·프로필 등록 · 신뢰도 링/점수 · 탐색 피드

현재 상태 — 사용자 여정 목업(🧭)에 있던 온보딩·프로필·신뢰도 링 화면을 데모(💬)에도 이식해, 최초 실행 시 온보딩 모달에서 관심사 태그·자기소개를 입력하면 신뢰도 링이 채워지고 탐색 피드에서 신뢰도 점수 기준 자동 승인까지 클라이언트 사이드 로직으로 재현했습니다. 신뢰도 점수의 체크인율·노쇼율 가중합 산식과 탐색 피드의 서버 연동은 아직 없습니다.

  1. MVP최초 대화 온보딩 — 봇 최초 실행 시 관심사 태그 버튼 선택 + 자기소개 한 줄을 받아 프로필 DB에 저장하고, 자기소개 텍스트는 매칭엔진 층위 2 임베딩 파이프라인에 그대로 재사용한다. 데모에서 검증 완료.
  2. 베타신뢰도 점수 산식 — 체크인율·노쇼율·참여횟수 가중합으로 시작(규칙 기반). 계산에 필요한 체크인 데이터가 베타 기간에야 쌓이므로 MVP 단계엔 배포할 수 없다. 데모의 신뢰도 링은 온보딩·체크인 이벤트에 점수를 더하는 단순 버전이다.
  3. 정식출시탐색 피드 — "이번 주 모집 중인 모임" 카드 리스트를 발송(부록에서 확인된 카드형 메시지 UI 활용), 신뢰도 점수를 정렬·필터 기준으로 노출한다. UI는 데모에서 검증 완료, 서버 연동은 별도.
계획서 조사만 존재

③ 실제 메신저(잔디·네이버웍스) 연동 · 예약/캘린더 실동작

현재 상태 — 부록에서 3사 봇 API를 문서 기준으로 실사했지만, 실제 봇 앱 등록·Webhook 수신 서버·OAuth 인증 코드는 아직 없습니다. 데모·랜딩은 모두 브라우저 시뮬레이션입니다.

  1. MVP네이버웍스 봇 등록 — 개발자센터에서 봇 앱 생성 → Webhook URL 발급 → 콜백 수신 서버(자연어 파싱)와 OAuth 2.0 토큰 발급/갱신 로직을 구현한다. 실사 결과 자유 텍스트 콜백이 되는 유일한 채널이라 MVP는 네이버웍스 단일 타깃으로 좁힌다.
  2. 베타캘린더 자동 등록 — 확정된 모임을 참가자 캘린더에 등록(제목·시간·장소). 캘린더 API는 이번 실사 범위 밖이라 베타 착수 전 별도 실사가 필요하다.
  3. 거래검증네이버 예약 연동 — 파트너센터 심사가 전제라 MVP는 "예약 링크 제공"으로 축소하고, 실연동은 04절의 예약 제휴 커미션과 같은 시점에 착수한다. 카카오워크는 모달 폼 개설로 변형 구현, 잔디는 주최 측 API 협의가 전제된다.

⚠ 네이버웍스 봇은 방 생성 시점에만 멤버를 지정할 수 있어(부록 실사 결과), 인접 슬롯 병합 시 방 재생성이 필요해 기존 설계보다 API 호출이 늘어난다.

데모에서 실동작 검증 완료

④ 방폭 (72시간 미달 자동 해산 + 전액 환급)

현재 상태 — 랜딩 데모(🤝)에서 타임어택 가속 버튼으로 실제 트리거해 "방폭 → 디파짓 전액 환급 → 안내 메시지" 시퀀스가 상태머신으로 재현됨을 검증했습니다. 다만 이는 클라이언트 사이드 타이머와 가상 포인트로 만든 시뮬레이션입니다.

  1. MVP서버 사이드 타이머 — 방 생성 시 마감 시각(생성+72h)을 DB에 기록하고 배치 잡(cron 또는 지연 큐)이 주기적으로 미달 방을 스캔·해산한다. 데모의 상태 전이 로직을 그대로 서버로 이관한다.
  2. MVP인접 슬롯 병합 제안 — 방폭 확정 전 매칭엔진이 인접 슬롯 후보를 조회해 병합 제안 카드를 먼저 발송하고, 미응답 시에만 방폭을 확정한다.
  3. 정식출시실결제 환급 — 가상 포인트가 아니라 조직 사내 복지포인트 시스템(있는 경우) 또는 자체 포인트 원장과 연동해 실제 잔액을 환급하고, 회계 감사를 위한 트랜잭션 로그를 남긴다.
MVP(가상 크레딧 원장) 실동작 검증 완료

⑤ 법인카드 집행·스폰서십 · HR 대시보드 연동

현재 상태 — 스폰서 셀프서브 목업과 별도 HR 대시보드 목업(📊)은 여전히 실제 결제·전자결의·조직 데이터 파이프라인이 없습니다. 다만 MVP 대안인 가상 크레딧 원장(회사 선결제 충전 → 모임 확정 시 차감 → 노쇼 시 몰수 / 정상 참석 시 전액 환급+보너스)은 데모(💬)에서 크레딧 잔액이 실제로 오가는 것까지 실동작으로 구현·검증했습니다. PG 결제 연동과 HR 대시보드 실연동은 아직 없고, "법인카드 자동 집행" 토스트는 여전히 목업 연출입니다.

  1. MVP가상 크레딧 원장 — 법인카드 실연동 대신, 회사가 선결제로 크레딧을 충전하고 모임 확정 시 차감되는 자체 원장으로 우회한다. 조직별 ERP·전자결의 시스템은 표준 API가 없어 커스텀 연동이 불가피하기 때문이다. 데모에서 검증 완료.
  2. 거래검증스폰서 결제 PG 연동 — PG사 결제 + 사업자 인증 API로 셀프서브 온보딩을 구현하고, 매칭 확정 카드에 스폰서 배지를 삽입하는 매칭엔진 훅을 추가한다.
  3. 플랫폼확장 이후HR 대시보드 실연동 — 매칭엔진의 체크인·노쇼·부서조합 로그를 집계 배치로 대시보드 API에 공급한다. 조직 HRIS와의 SSO·구성원 매핑이 필요하나 부서 매핑은 조직도 API로 이미 확인됐다. 법인카드 실연동은 조직별 개별 계약 건으로 Enterprise 단계에서만 대응한다.
공통 패턴 — 다섯 항목 모두 UI·로직 설계는 끝났고 외부 인프라(서버 배치, 결제 PG, 캘린더/예약 API, 조직 HRIS)를 붙이는 일만 남았다는 공통점이 있습니다. ①②④⑤ 네 항목은 데모·랜딩에서 클라이언트 사이드 실동작 검증까지 마쳤고, 실메신저 연동(③)만 API 실사 단계로 남아 있습니다. MVP 범위(~2026.08)는 서버 사이드 방폭 배치, 체크인·온보딩·크레딧 원장의 서버 API 이관, 네이버웍스 봇 등록까지이고, 나머지는 06절 실행 계획의 베타·정식출시·거래검증 단계에 맞춰 순서대로 붙입니다.

기획 단계에서 잔디·카카오워크·네이버웍스의 공식 개발자 문서를 항목별로 실사했습니다. "3사 동시 지원"을 가정하지 않고, 실제 API가 허용하는 범위로 MVP를 설계합니다.

실사 결론 — MVP는 네이버웍스 단일 타깃 — 간판 기능인 자연어 한 줄 개설(층위 1 AI)이 실제로 가능한 곳은 자유 텍스트 콜백을 지원하는 네이버웍스뿐입니다. 카카오워크는 모달 폼 기반 변형으로 2순위 확장하고(방 생성·초대 API는 오히려 가장 유연), 잔디는 주최 측과 개발용 API 협의를 전제로 로드맵에 둡니다. 방 생성 시점에만 멤버를 지정할 수 있는 네이버웍스 제약은 "매칭 확정 후 방 개설·인접 슬롯 병합 시 새 방 재생성"이라는 기존 플로우 설계와 자연스럽게 맞습니다.

같은 공간에서 일해도 낯선 사람으로 남는 조직에게, OfficeMate는 매 끼니를 관계로 바꾸는 가장 작은 개입입니다.
